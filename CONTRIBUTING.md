# Contributing to Bicep PowerShell

You are more than welcome to contribute to the Bicep PowerShell module, whether it is [Pull Requests](#pull-requests), [Feature Suggestions](#feature-suggestions) or [Bug Reports](#bug-reports)!

## Getting Started

### Prerequisites

Before you begin, make sure you have the following installed:

- **PowerShell 7+** – The module targets PowerShell Core. Download from [GitHub](https://github.com/PowerShell/PowerShell/releases).
- **.NET SDK 8.0** – Required to build the C# projects. The exact patch version is pinned in `global.json`. Download from [dotnet.microsoft.com](https://dotnet.microsoft.com/download).
- **Git** – For cloning the repository and submitting pull requests.

### Initial setup

1. Fork this repo (see [this forking guide](https://guides.github.com/activities/forking/) for more information).
2. Checkout the repo locally:
   ```powershell
   git clone git@github.com:{your_username}/PSBicep.git
   cd PSBicep
   ```
3. Download all build dependencies and build the module:
   ```powershell
   .\build.ps1 -ResolveDependency -Task build
   ```
   This installs the required PowerShell modules into `output/RequiredModules`, compiles the C# projects, and packages the final module into `output/Bicep/<version>/`.

4. Import the built module to verify everything works:
   ```powershell
   Import-Module ./output/Bicep
   Get-BicepVersion
   ```

## Repository Structure

The repository is organized as follows:

```
PSBicep/
├── .build/tasks/          # Custom Invoke-Build task definitions
├── .github/               # GitHub Actions workflows and templates
├── Docs/Help/             # Markdown help files for all public commands
├── PSBicep/               # C# project – assembly loader (nested module)
├── PSBicep.Core/          # C# project – thin wrapper around Bicep assemblies
├── Source/                # PowerShell module source code
│   ├── Assets/            # Generated BicepTypes.json (created by build)
│   ├── Classes/           # PowerShell classes and enums
│   ├── en-US/             # XML external help files (generated by build)
│   ├── Private/           # Internal PowerShell helper functions
│   ├── Public/            # Exported PowerShell cmdlets (24 commands)
│   ├── Bicep.psd1         # Module manifest
│   ├── Bicep.psm1         # Root module file
│   └── suffix.ps1         # Appended after module merge by ModuleBuilder
├── Tests/                 # Pester test files
│   └── supportFiles/      # Bicep/JSON/ARM templates used by tests
├── scripts/               # Helper scripts for GitHub Actions and local dev
├── build.ps1              # Main entry point for the build system
├── build.yaml             # Sampler/ModuleBuilder pipeline configuration
├── global.json            # Pins the .NET SDK version
├── GitVersion.yml         # Automatic semantic versioning configuration
├── PSBicep.sln            # Visual Studio solution file
└── RequiredModules.psd1   # PowerShell module build dependencies
```

### Architecture: Three-Tier Design

The module is split into three layers to isolate Bicep's assemblies from the rest of the PowerShell session and avoid DLL conflicts:

#### Tier 1 – PSBicep.Core (C#)

Located in `PSBicep.Core/`. This is a thin C# wrapper around the official Bicep NuGet packages. It exposes a clean API (`BicepService`, `BicepHelper`, etc.) that the upper layer calls. All Bicep assemblies are pulled in here as NuGet references.

Key source files:

- `BicepService.cs` / `Services/BicepCoreService.cs` – Core Bicep operations (build, decompile, format, publish, …)
- `Services/BicepRegistryService.cs` – Module registry (publish/restore)
- `Models/` – Data transfer objects returned to PowerShell
- `Authentication/` – Azure authentication helpers
- `Rewriters/` – Bicep AST rewriters (e.g. removing unknown properties on export)
- `Logging/` – Diagnostic logging bridge

#### Tier 2 – PSBicep (C#)

Located in `PSBicep/`. This project is loaded as a PowerShell binary module (nested module). Its only job is to load all Tier 1 assemblies into a custom [Assembly Load Context](https://learn.microsoft.com/dotnet/core/dependency-loading/understanding-assemblyloadcontext) so they do not conflict with assemblies already loaded by other modules in the session.

Key source files:

- `LoadContext/DependencyAssemblyLoadContext.cs` – The isolated load context
- `LoadContext/BicepLoader.cs` – Discovers and loads assemblies from `PSBicep.Core/bin/`
- `Commands/` – C# `PSCmdlet` implementations that call into `PSBicep.Core` (e.g. `BuildBicepFileCommand.cs`, `PublishBicepFileCommand.cs`)

#### Tier 3 – Source (PowerShell)

Located in `Source/`. This is where most contributors will spend their time. It contains PowerShell wrappers around the C# commands as well as purely PowerShell-based cmdlets.

- **`Source/Public/`** – The 24 exported cmdlets. Each file contains one function.
- **`Source/Private/`** – Internal helper functions (validation, Azure helpers, markdown generators, …).
- **`Source/Classes/`** – PowerShell classes such as `BicepDiagnosticEntry` and tab-completion argument completers.
- **`Source/Assets/`** – `BicepTypes.json` downloaded by the `getBicepTypes` build task. Used by commands that need to know the full set of ARM resource types.
- **`Docs/Help/`** – Markdown help files, one per cmdlet. Edited by contributors and converted to XML by the build.

## Build System

The build system is powered by [Invoke-Build](https://github.com/nightroman/Invoke-Build) and the [Sampler](https://github.com/gaelcolas/Sampler) framework. All tasks are declared in `build.yaml` and custom tasks live under `.build/tasks/`.

### Common build commands

| Goal | Command |
|---|---|
| First-time setup + full build | `.\build.ps1 -ResolveDependency -Task build` |
| Rebuild after changing PowerShell source | `.\build.ps1 -Task build` |
| Fast iteration (skips help generation) | `.\build.ps1 -Task quickbuild` |
| Compile C# only | `.\build.ps1 -Task compile` |
| Run tests | `.\build.ps1 -Task test` |
| Package as NuGet | `.\build.ps1 -Task pack` |

### Build task reference

| Task | Description |
|---|---|
| `PSBicep.Restore` | Runs `dotnet restore` on the C# solution |
| `PSBicep.Compile` | Runs `dotnet publish` for `PSBicep` and `PSBicep.Core`; outputs DLLs to `PSBicep/bin/` and `PSBicep.Core/bin/` |
| `Build_Module_ModuleBuilder` | Merges all PowerShell source files into a single `.psm1` and copies assets |
| `getBicepTypes` | Downloads `BicepTypes.json` from GitHub into `Source/Assets/` |
| `updateMarkdownHelp` | Regenerates markdown from comment-based help using platyPS |
| `repairPlatyPSHelp` | Fixes known formatting issues in platyPS output |
| `updateExternalHelp` | Converts markdown in `Docs/Help/` to XML help files in `Source/en-US/` |
| `Pester_Tests_Stop_On_Fail` | Runs the Pester test suite and stops on the first failure |

### Updating the Bicep version

The pinned Bicep CLI version is set in `RequiredModules.psd1` (`'Azure/Bicep'`) and the NuGet package version is referenced in `PSBicep.Core/PSBicep.Core.csproj`. Both must be updated together when bumping Bicep.

## Development Workflows

### Working on a PowerShell cmdlet

1. Edit or add the function in `Source/Public/` (for exported commands) or `Source/Private/` (for internal helpers).
2. Update the corresponding markdown help file in `Docs/Help/`. If adding a new command, create a new markdown file based on an existing one.
3. Build and test:
   ```powershell
   .\build.ps1 -Task quickbuild
   Import-Module ./output/Bicep -Force
   # Test your changes manually, then run the relevant Pester tests:
   .\build.ps1 -Task test
   ```

For rapid iteration you can also import directly from source (no C# recompile needed):

```powershell
Import-Module .\Source\Bicep.psd1 -Force
```

> **Note:** Importing from source requires the C# assemblies to already be compiled. Run `.\build.ps1 -Task compile` once after cloning.

### Working on C# code (PSBicep or PSBicep.Core)

1. Edit the relevant `.cs` file(s) inside `PSBicep/` or `PSBicep.Core/`.
2. Recompile and rebuild the module:
   ```powershell
   .\build.ps1 -Task compile   # recompile C# only
   .\build.ps1 -Task quickbuild # or rebuild the full module quickly
   Import-Module ./output/Bicep -Force
   ```
3. Run the tests:
   ```powershell
   .\build.ps1 -Task test
   ```

You can also open `PSBicep.sln` in Visual Studio or Rider for a richer C# editing experience. The `global.json` file pins the .NET SDK version to ensure consistent builds.

## Help Documentation

[platyPS](https://github.com/PowerShell/platyPS) is used to author external help in Markdown. A slightly modified fork of platyPS is pulled in by the build script (`SimonWahlin/platyPS` in `RequiredModules.psd1`).

When adding or changing a cmdlet:

1. Edit or create the corresponding file in `Docs/Help/` (e.g. `Docs/Help/Build-Bicep.md`).
2. Fill in any `{{ ... }}` placeholder sections the build generates.
3. Run `.\build.ps1 -Task build` to regenerate the XML help files in `Source/en-US/`.

## Tests

[Pester](https://github.com/pester/Pester) (v5) is the test framework. Tests live under `Tests/` and follow the naming convention `<CommandName>.Tests.ps1`.

### Running tests

```powershell
# Run the full test suite
.\build.ps1 -Task test

# Run tests for a specific tag
.\build.ps1 -Task test -PesterTag Unit

# Exclude slow or integration tests
.\build.ps1 -Task test -PesterExcludeTag Integration
```

Test results are written to `output/testResults/` in NUnit XML format.

### Writing tests

- Each public command should have a corresponding test file in `Tests/`.
- Use files in `Tests/supportFiles/` as fixtures (sample `.bicep`, ARM JSON, parameter files, …).
- Follow the existing Pester `Describe`/`Context`/`It` block style used in the other test files.

## Pull Requests

If you like to start contributing to Bicep PowerShell. Please make sure that there is a related issue to link to your PR.

- All PRs should be tagged using the labels, `newFeature`, `bugFix`, `updatedDocs` or `enhancement`. This is used for automatic release notes creation.
- Make sure that the issue is tagged in the PR.
- Write a short but informative commit message, it will be added to the release notes.

## Feature Suggestions

- Please first search [Open Issues](https://github.com/PSBicep/PSBicep/issues) before opening an issue to check whether your feature has already been suggested. If it has, feel free to add your own comments to the existing issue.
- Ensure you have included a "What?" - what your feature entails, being as specific as possible, and giving mocked-up syntax examples where possible.
- Ensure you have included a "Why?" - what the benefit of including this feature will be.
- Use the "Feature Request" issue template [here](https://github.com/PSBicep/PSBicep/issues/new/choose) to submit your request.

## Bug Reports

- Please first search [Open Issues](https://github.com/PSBicep/PSBicep/issues) before opening an issue, to see if it has already been reported.
- Try to be as specific as possible, including the version of the Bicep PowerShell module, PowerShell version and OS used to reproduce the issue, and any example files or snippets of Bicep code needed to reproduce it.
- Use the "Bug Report" issue template [here](https://github.com/PSBicep/PSBicep/issues/new/choose) to submit your request.
